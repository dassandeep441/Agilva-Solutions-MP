Overview

Agilva Solutions is a certified Atlassian Marketplace vendor. This Security Policy describes two things: the security requirements we meet as an Atlassian partner, and the internal practices we follow when building, shipping, and maintaining QR Hub for Jira.

QR Hub for Jira is built entirely on Atlassian Forge — Atlassian's serverless, sandboxed app platform. This architectural choice means the vast majority of infrastructure-level security responsibilities (compute isolation, network security, physical security, DDoS protection) are inherited directly from Atlassian's own enterprise-grade platform, not managed by Agilva independently.

🔒 Because QR Hub for Jira runs entirely on Forge with no external servers, APIs, or data egress, its attack surface is substantially smaller than traditional cloud apps. There is no Agilva-controlled backend to breach.

Atlassian Partner Security Requirements

All Atlassian Marketplace vendors must comply with Atlassian's Marketplace Partner Agreement and associated security standards. The key requirements and how QR Hub for Jira meets each are described below.

🏗️

Forge-Only Architecture

Apps built on Forge must not exfiltrate data outside Atlassian's infrastructure. QR Hub has no external server, no webhook, and no third-party API call of any kind.

🔐

Minimal Permission Scopes

Atlassian requires partners to request only the permissions necessary for app function. QR Hub requests the minimum scopes needed to read issue data for QR generation.

📋

Privacy Policy Publication

Partners must publish a clear, accessible privacy policy. Agilva's privacy policy for QR Hub is available at privacy.html and linked from the Marketplace listing.

🔄

Vulnerability Disclosure

Partners must have a process for receiving and responding to security vulnerability reports. Agilva's security contact is documented below and monitored daily.

📦

Dependency Hygiene

Partners must keep third-party dependencies up to date and free of known critical vulnerabilities. Agilva performs regular dependency audits before every release.

🛡️

Data Residency Compliance

Forge apps automatically inherit Atlassian's data residency controls. Forge Storage data is stored within the same region as the customer's Atlassian site.

Secure Development Practices

Agilva follows a security-first development approach throughout the software development lifecycle (SDLC) for all Marketplace apps.

Code review

All code changes require a peer review before merging. No code is deployed to production directly from a single developer's branch. Reviews explicitly check for: injection risks, unsafe data handling, unnecessary permission use, and insecure defaults.

Security testing

Each release includes:

  • Static analysis via ESLint with security-focused rule sets
  • Dependency vulnerability scanning via npm audit prior to every release
  • Manual security review against the OWASP Top 10 for any feature touching user input or stored data

Forge security model

QR Hub for Jira is built using Atlassian's Forge platform, which enforces a strict security sandbox:

  • Each app execution runs in an isolated runtime — no shared state between tenants
  • Outbound network calls are restricted by Forge's egress controls; QR Hub makes none
  • All Forge Storage reads and writes are scoped to the installing tenant only
  • Atlassian manages runtime patching, OS hardening, and compute isolation

Access Control

Internal access

Production Forge environments are managed exclusively through the Atlassian Developer Console. Access is restricted to authorised Agilva engineers and protected by Atlassian's own SSO and MFA requirements.

No Agilva-controlled servers

There are no Agilva-managed servers, databases, or cloud infrastructure associated with QR Hub for Jira. There is therefore no Agilva-side access control surface for customer data — because no customer data reaches Agilva systems in the first place.

App permissions within Jira

QR Hub for Jira respects all existing Jira permission schemes. The app reads issue data only for the purpose of generating the QR code displayed in the panel. It does not modify issues, projects, or any Jira configuration. Users who lack access to an issue in Jira will not have that access granted by the app.

Data Handling & Retention

QR Hub for Jira processes the minimum data necessary to generate QR codes. It does not build user profiles, track behaviour, or aggregate data across tenants.

What is stored

The only data persisted by QR Hub for Jira is the URL entered into the Dashboard Gadget, stored via storage:app in Atlassian Forge Storage. This URL is stored per-site, not per-user, and persists only until the gadget is removed or the URL is updated.

What is not stored

  • Issue content, summaries, or field values
  • User identities or account IDs
  • QR code images (generated in-browser, never sent to any server)
  • Usage analytics or telemetry of any kind

Retention & deletion

Stored gadget URLs are automatically deleted when the app is uninstalled from a Jira site. Atlassian Forge Storage provides no mechanism for Agilva to access or export individual tenants' stored data outside of the app's own runtime context.

Dependency Management

QR Hub for Jira is built on Atlassian Forge with a deliberately lean dependency tree. We maintain security of third-party code as follows:

  • Audit before every release: npm audit is run as part of the pre-release checklist. Releases are blocked if any high or critical severity vulnerabilities are detected.
  • Regular dependency updates: Dependencies are reviewed and updated at minimum once per month, and immediately upon disclosure of a relevant CVE.
  • Minimal footprint: We avoid pulling in large general-purpose libraries where a smaller, purpose-built alternative or a native Forge API achieves the same result.

Incident Response

In the event of a confirmed security incident affecting QR Hub for Jira, Agilva will:

  1. Assess within 24 hours of detection or report — determine scope, affected tenants, and severity.
  2. Contain — if the issue is in Forge-controlled infrastructure, coordinate with Atlassian's security team. If in Agilva-controlled code, deploy a patch or temporarily disable the affected feature.
  3. Notify affected customers within 72 hours of confirming a breach that affects their data, in line with GDPR Article 33 obligations where applicable.
  4. Post-incident review — document the root cause, timeline, and remediation steps. Update internal controls to prevent recurrence.

🚨 Given that QR Hub for Jira stores no personally identifiable information and no sensitive business data, the practical risk and scope of any incident is inherently limited.

Atlassian Forge Platform Security

By building on Forge, QR Hub for Jira inherits a comprehensive set of platform-level security controls maintained by Atlassian. These include:

  • Compute isolation: Each app invocation runs in a fully isolated sandbox. No cross-tenant code execution is possible.
  • Encrypted storage: All data in Forge Storage is encrypted at rest using AES-256 and in transit using TLS 1.2+.
  • SOC 2 Type II & ISO 27001: Atlassian's infrastructure — including Forge — is independently audited and certified. Certificates are available on atlassian.com/trust.
  • DDoS & network protection: Managed by Atlassian at the platform level. No Agilva configuration required or possible.
  • Bug Bounty Programme: Atlassian operates an active bug bounty programme covering the Forge platform itself.

Full details of Atlassian's security programme are available at atlassian.com/trust/security.

Security Contact

To report a security vulnerability, suspected breach, or raise a security concern related to QR Hub for Jira, please contact Agilva directly:

💋 Security disclosures: Contact us via the form and select General / Other as the topic, noting "Security Disclosure" in your message. We treat all security reports as high priority and aim to respond within one business day.

We follow responsible disclosure practices. If you report a valid vulnerability, we will work with you to resolve it before any public disclosure, and credit your report in our release notes if you wish.
